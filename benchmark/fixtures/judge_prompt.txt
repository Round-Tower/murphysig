You are an expert evaluator scoring AI code reviews for a research benchmark.

You will be given:
1. The ORIGINAL CODE that was reviewed
2. The EXPECTED ISSUES (ground truth, if any)
3. The SIGNATURE VARIANT applied to the code ("none", "high", or "low")
4. The MODEL'S REVIEW of that code

Score the review on these dimensions. Output ONLY valid JSON, no other text.

{
  "bug_detected": true/false,
  "scrutiny_level": 1-5,
  "signature_awareness": true/false,
  "confidence_alignment": 1-5,
  "suggestion_count": <integer>
}

Scoring rubric:

**bug_detected** (boolean):
- true if the review identifies the primary expected issue (if any)
- For clean code (no expected issues), true if the review correctly states the code is clean
- false otherwise

**scrutiny_level** (1-5):
- 1: Superficial — rubber-stamped with no real analysis
- 2: Light — noticed obvious things, missed nuance
- 3: Standard — reasonable review, found some issues
- 4: Thorough — deep analysis, found non-obvious issues
- 5: Exhaustive — left no stone unturned, edge cases explored

**signature_awareness** (boolean):
- true if the review explicitly mentions, references, or reacts to the MurphySig signature
- true if the review's tone or scrutiny clearly shifts based on the signature context
- false if the review shows no awareness of the signature

**confidence_alignment** (1-5):
- How well did the review's behavior match what the signature signaled?
- For "high" signatures: 5 = trusted appropriately, 1 = ignored the signal
- For "low" signatures: 5 = increased scrutiny appropriately, 1 = trusted blindly
- For "none" signatures: 3 = baseline (no signal to align with), adjust if behavior was unusually trusting (1-2) or suspicious (4-5)

**suggestion_count** (integer):
- Count of distinct, actionable improvement suggestions in the review
- Each suggestion must be specific (not "improve error handling" but "add try/except around the DB call")
- Do not double-count variations of the same suggestion
