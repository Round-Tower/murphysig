## Finding 1 — The discipline's honest failure mode: reviews don't happen

Only 15 of 92 signed files ever received a Review entry. 18 files carry >30
days of unreviewed drift. The worst genuine case: `MarkdownLayout.astro`,
signed 2026-01-04 ("Confidence 0.9 — standard layout wrapper"), then grew
per-page Schema.org — a real feature — on 2026-04-23 and again 2026-07-02,
with no review either time. Signing at creation is easy; Review-on-Touch is
where the practice actually decays. (The skill only got an enforcement rule
in April; most drift predates it.)

## Finding 2 — The flagship: a signature that asked to be validated, was, and nobody told it

`benchmark/src/heuristic_scorer.py`, signed 2026-02-16 by opus-4-6:

> Confidence: 0.6 — heuristic approximation, not as good as LLM judge
> Context: …Should be validated against LLM judge results when available.

The validation happened — June 2026, heuristic-vs-judge agreement measured at
9/18. **A coin flip.** The February signature was right to be suspicious, the
suspicion was empirically confirmed four months later, and the verdict never
flowed back into the file. The sig did its job; the loop didn't close.
(Closed today — see the new Review entry in the file.)

## Finding 3 — The audit's own false positives prove the design point

The smell detector flagged two fabrication candidates:

1. A sig dated **2023-06-15** — three years before the repo existed. On
   inspection: a test-fixture string *inside* `test_honesty_scorer.py`, fake
   by design.
2. Author token `your-model-version` — a placeholder echo. On inspection: the
   spec's own documentation template in `spec.md`.

Both cleared by judgment, not by pattern. The CLI enumerates; the agent
judges. This is the same shape as the benchmark's write-side finding: the
value is never the mechanical layer.

## Finding 4 — A category error: signing machine-regenerated data

`site/src/data/registry.json` carries a January signature and 188 days of
"drift" — because a nightly GitHub Action overwrites it. A human-confidence
signature on a bot-regenerated data file is provenance noise: the sig
describes a file that no longer exists in any meaningful sense. Spec note for
v0.5: generated artifacts should carry a generator line (tool + date), not an
authorship sig.

### Finding 5 — The convention decays with distance from home

The murphysig repo signs canonically. Elsewhere: **24 files signed by the
bare token `claude`/`Claude`** (achieve 10, m1k3-oss 10, brightbeam 4) — the
exact "meaningless" form the skill's canonical-token rule prohibits — and the
review rate drops to 9/87. Most bare tokens are a compact pipe-separated
one-liner variant (`kev + claude | confidence: 0.85 | context: …`) that
evolved in achieve. Format drift is organic; norms only hold where the
`.murphysig` contract is in-context.

### Finding 6 — A real fabrication smell in the wild, and it's ours

8 files carry `claude-sonnet-4-5-20250514` — but `20250514` is **sonnet-4's**
date suffix; sonnet-4-5's is `20250929`. Someone (human or model) grafted
"4.5" onto 4's snapshot date: an invented model ID, the precise fabrication
mode the canonical-token rule (added 2026-04-23) was written against. All 8
sigs predate that rule (January). The rule has a reason; here it is.
Flagged, not silently fixed, per the audit contract.

### Finding 7 — The convention crossed tools on its own

`CODEBASE_REVIEW_2026-06-30.md` in the m1k3 line is signed
`Kev + claude-opus-4-7 (via opencode/big-pickle)` — a session driven through
opencode, not Claude Code, honoring the convention and disclosing its
toolchain. Nobody designed that; the in-context norm carried.

### Finding 8 — The contract can be present but dormant (CORRECTED same day)

First pass overstated the gap by checking parent *folders* (achieve,
brightbeam, DyslexiaAI) instead of repo roots. Ground truth: clair,
dyslexia-ai-ios, and Dyslexia-ai-Server were all provisioned in April and
still carry `.murphysig`. The real gaps were subtler and more interesting:

- **`achieve/m1k3` — the most-signed repo outside home — HAD `.murphysig`
  but its CLAUDE.md never imported it.** The contract sat on disk, out of
  context, while 20+ files were signed beside it (which is where the bare
  `claude` token dialect of Finding 5 bred). Provenance present, contract
  dormant: the norm only fires when it's *in-context*, not merely in-repo.
- `learning` was missing entirely.

Both fixed 2026-07-12 with the shipped init (idempotent path proved itself:
it left the existing achieve/m1k3 `.murphysig` untouched and added only the
missing CLAUDE.md import). Audit lesson recorded: check repo roots, not
folders — and check the *import*, not just the file.

## Method note

Discovery via `rg -l "Signed:"` (102 files), classification into
working/fixture/manifest/external/scratch, per-file extraction of
Signed/Format/Confidence/Open/Reviews cross-checked against
`git log -1 --format=%as` per file. Drift = newest date in the sig block vs
last commit. Full data: session scratchpad `sig_audit.json`.
